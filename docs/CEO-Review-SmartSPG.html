🏠 1. Landing Page

La landing page tiene un diseno profesional y moderno. Hero con animacion de streaming IA, seccion "About Us", social proof, "How It Works", grid de features, tabla de precios, FAQ e integraciones.

Propuesta de valor clara
"Propuestas comerciales multi-servicio, generadas con IA en minutos" — comunica el valor inmediatamente. El badge "Powered by Claude AI" genera confianza tecnica.
Pricing bien estructurado (3 tiers)
Starter (gratis 30 dias), Pro $49/mes (60 dias gratis — "popular"), Enterprise (contactar). El modelo freemium 30+30 es muy atractivo para LATAM.
Inconsistencia en dias de prueba
El Hero dice "60 dias sin tarjeta" pero Starter ofrece 30 dias sin tarjeta y Pro ofrece 60 (30 sin tarjeta + 30 con tarjeta). Esto puede confundir al usuario sobre que plan obtiene realmente.
Enterprise CTA lleva a mailto:hola@smartspg.io
El dominio smartspg.io no esta configurado aun. Esto dara error al usuario si intenta contactar ventas.
Idioma 100% espanol
Excelente para el mercado LATAM target. Sin embargo, "Billing" en la navegacion lateral aparece en ingles — deberia ser "Facturacion" o "Plan".

🔐 2. Sign-In / Sign-Up

Autenticacion implementada con Clerk. Layout split-screen profesional: panel izquierdo con branding y trust items, panel derecho con formulario Clerk.

Branding consistente en auth
Panel lateral con logo SmartSPG, trust items ("60 dias gratis", "Export PDF y Word", "Multi-servicio B2B"), y mensaje persuasivo. Muy profesional para SaaS B2B.
Flujo con Organizations de Clerk
Despues de sign-up redirige a select-org, donde el usuario crea su organizacion. Esto habilita multi-tenancy real desde el dia 1.
Sin validacion de dominio corporativo
Cualquier email puede registrarse. Para B2B LATAM seria valioso validar dominios corporativos o al menos sugerir email de empresa vs. Gmail.

🚀 3. Onboarding (5 pasos)

Wizard de 5 pasos con chips seleccionables: tipo de negocio, industrias + servicios, tamano de propuestas, tono preferido, y confirmacion.

Onboarding que personaliza la experiencia
Los datos de onboarding se guardan en metadata JSONB del tenant. Esto permite personalizar templates, tonos y sugerencias para cada empresa.
BUG: Validacion de onboarding comentada
En onboarding/page.tsx, la verificacion de si el onboarding ya fue completado esta comentada (lineas 18-26). Un usuario puede acceder al onboarding repetidamente o quedar atrapado en un loop.
Onboarding no se puede saltar
No hay boton "Saltar" o "Completar despues". Un CEO impaciente podria abandonar el producto aqui. Recomendacion: agregar opcion de skip con valores default.

📊 4. Dashboard

Dashboard principal con saludo personalizado, 4 stat cards, tabla de propuestas recientes, y charts (donut + barras).

Dashboard visualmente atractivo
Saludo contextual (Buenos dias/tardes/noches + nombre), stat cards con tendencias, tabla de propuestas recientes con badges de status, y graficos Recharts.
100% datos mock hardcodeados
Las 4 stat cards, la tabla de propuestas y los graficos usan datos ficticios estaticos. Un CEO que entre vera siempre "48 propuestas, $93.5K en valor" independientemente de su actividad real. Esto es CRITICO corregir antes de beta.
CTA "Nueva propuesta" necesita mas prominencia
El boton esta arriba a la derecha pero el flujo principal del producto es generar propuestas. Considerar un empty state con CTA central cuando no hay propuestas.

✨ 5. Wizard de Propuestas (Core del Producto)

Wizard de 4 pasos: seleccionar cliente, definir contexto, generacion IA con streaming, revision y export.

Paso 1 — Seleccion de Cliente

Busqueda de clientes con debounce
Busca clientes existentes via API con skeleton loading. Permite crear cliente inline. UX fluida.

Paso 2 — Contexto

Multi-servicio bien implementado
Chips seleccionables para servicios (Paid Media, Landing Pages, Diseno, etc.), slider de budget, datepicker de timeline, selector de tono. Diferenciador clave vs competencia.

Paso 3 — Generacion IA

Streaming con UX premium
Panel oscuro (#0F172A) con progress bar, iconos animados por seccion (pending/streaming/done), preview de texto generado. Comunicacion clara: "Claude esta generando..." con estimado de 30 segundos.
BUG CRITICO: Schema mismatch entre cliente y servidor
El schema del servidor (route.ts) define campos: serviciosPropuestos, alcancePorServicio, casoDeExito.
El schema del cliente (Step3Generate.tsx) espera campos: solucion, alcance.
Esto significa que la IA genera campos que el frontend no mapea correctamente. El usuario podria ver secciones vacias o incompletas.
Sin retry en caso de error de IA
Si la llamada a Claude falla, se muestra un error pero no hay boton de reintentar. El usuario debe volver atras y avanzar de nuevo.

Paso 4 — Revision y Export

Editor TipTap con export triple
Secciones editables con rich text, export a PDF (DocuForge), DOCX y Email. Muy completo para un MVP.

👥 6. Gestion de Clientes

Tarjetas con avatar gradiente, busqueda con debounce, filtros por industria, y "Opportunity Score".

Cards visualmente ricas
Cada cliente muestra nombre, empresa, industria como tag, score de oportunidad con barra de progreso, y conteo de propuestas. Diseno limpio y util.
Boton "Nuevo Cliente" sin funcionalidad
El boton esta renderizado pero no abre ningun formulario ni modal. Un CEO que intente agregar un cliente se frustrara.
Sin vista de detalle de cliente
No hay pagina individual de cliente con historial de propuestas, notas, o edicion de datos. Feature esencial para CRM basico.

💳 7. Billing y Trial

Sistema de trial hibrido: 30 dias sin tarjeta + 30 dias con tarjeta. Integracion con Stripe SetupIntent.

Trial gating implementado end-to-end
El stream de propuestas verifica trial status y devuelve 402 si la cuota se agoto. El billing page muestra dias restantes, propuestas usadas, y CTA contextual segun stage del trial.
Sin pagina de checkout para plan Pro
Cuando el trial expira, el boton dice "Activar plan Pro" pero no hay flujo de pago real implementado (Stripe Checkout o Payment Element). Solo existe SetupIntent para extender trial.

📈 8. Analytics

Dashboard analitico completo
6 KPIs, grafico de lineas (propuestas enviadas vs aceptadas), donut de status, barras de revenue por industria, tabla de top clientes. Muy impresionante para MVP.
Datos 100% ficticios (igual que Dashboard)
Todas las metricas son sample data hardcodeado. Si un CEO ve "tasa de cierre del 72%" sin haber creado una sola propuesta, pierde credibilidad inmediata.

⚙️ 9. Arquitectura y Seguridad

Multi-tenancy solida
tenant_id en todas las tablas, header X-Tenant-ID en todas las requests, middleware de Clerk que lo extrae del JWT. Patron correcto para SaaS B2B.
AI implementado correctamente
streamObject() en Route Handler (server-side), nunca en client. Cache de prompts con cacheControl ephemeral. Save non-blocking post-stream. Patron de referencia.
Stack moderno y mantenible
Next.js 16 App Router, Tailwind v4, shadcn/ui, Vercel AI SDK v4, Zod v4, DrizzleORM, Clerk, Stripe. Todas las librerias en sus versiones mas recientes.
Tokens y secrets expuestos en .git/config
El archivo .git/config contiene tokens de GitHub (ghp_...) y HuggingFace (hf_...) embebidos en las URLs de remote. Si este repo se comparte o se clona, los tokens quedan expuestos. ACCION URGENTE: remover tokens de remote URLs y usar variables de entorno o credential helpers.
Clerk Publishable Key hardcodeada en Dockerfile
La pk_test_ esta como ARG default en el Dockerfile. Aunque publishable keys no son secrets, es mejor practica pasarla solo como build arg en CI/CD.

💡 Veredicto Ejecutivo

SmartSPG tiene una base solida para un producto SaaS B2B. El diseno es profesional, la arquitectura multi-tenant es correcta, y el flujo core de generacion de propuestas con IA streaming es impresionante.

Sin embargo, hay 3 bloqueantes criticos antes de poder invitar usuarios beta: el schema mismatch que rompe el wizard IA, los datos mock en dashboard/analytics que destruyen credibilidad, y los tokens expuestos en git.

Estimacion para beta-ready: 2-3 dias de desarrollo enfocado en los P0 y P1 identificados. El producto tiene potencial real para el mercado LATAM de propuestas B2B.
